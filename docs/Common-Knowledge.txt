================================================================================
  PHẦN 1 — TỔNG QUAN MÔ HÌNH
================================================================================

Bài toán: IRS-assisted RSMA cho vệ tinh (NTN băng L, 1.58 GHz). Agent RL điều khiển
4 quyết định mỗi bước để tối đa reward = sum_rate − QoS_penalty.

Actor pipeline (4 stage/step):
  1. IRS selection (LƯỢNG TỬ): state → AE encoder → z_t → QC (nq qubit, L=3, data
     re-uploading) → post-NN → logits gán mỗi user vào {direct, IRS_1..M}.
  2. PhaseMLP: chọn pha IRS (N=24 phần tử/IRS × M, mức rời rạc 2-bit). Trọng số CHIA
     SẺ theo phần tử.
  3. PowerMLP: phân bổ công suất [w_c (common, theo group), w_p (private)] Σ=P_S.
  4. CkMLP: chia common-rate C_k trong group (softmax).

Critic: V(s) MLP (state-value). Dùng làm baseline cho advantage.
AE: dual-branch encoder (IRS branch + user micro-encoder chia sẻ theo user) → z_t;
    decoder tái tạo d_aff = K*(M+2)+2M (regulariser).
Lượng tử: tham số học được λ (encoding scales) + θ (variational angles), train bằng
    parameter-shift / SPSA. λ encoding: α=π·tanh(λ_y·z), δ=π·tanh(λ_z·z).
Thuật ngữ mạch: đây là VQC (data re-uploading) — PQC có tham số được tối ưu variational.


================================================================================
  PHẦN 2 — CƠ CHẾ CỐT LÕI (hiểu "tại sao")
================================================================================

REWARD:  reward = sum_rate − qp_penalty
  qp = λ_D · Σ_k (shortfall_k / (D_k+ε))²,  shortfall_k = max(0, D_k − R_tot_k)
  → reward RẤT NHẠY với QoS: 1 user fail → penalty lớn (∝ λ_D). reward âm sâu khi
    nhiều user dưới ngưỡng. Đây là "vòng xoáy" khi QoS tụt (penalty tăng → reward
    càng âm → advantage nhiễu → policy càng tệ).

GAE (gae_lambda): Â_t = Σ(γλ)^l δ_{t+l}. λ=0 → TD 1-bước (bias cao, KHÔNG credit dài
  hạn). λ→1 → Monte-Carlo. Phối hợp IRS/power/Ck là bài toán DÀI HẠN (200 bước) →
  cần λ cao (≈0.9) để credit đúng. λ=0 là nguyên nhân gốc của collapse (xem [A]).

CRITIC + WARM-UP: advantage chỉ tốt khi V(s) chính xác. Critic dựng mới (blind,
  explVar≈0) feed advantage rác cho actor đã tốt → phá policy. Warm-up = roll policy
  ĐÓNG BĂNG, fit V(s) theo Monte-Carlo return TRƯỚC PPO → V đúng thang từ ep 1.
  Critic được lưu kèm checkpoint → resume tự nạp lại. Đổi λ_D/R_LoS làm dịch thang
  return → critic_target_tau cao hơn giúp re-calibrate nhanh.

ENTROPY (β): khuyến khích explore. NHƯNG "β·H phải ≪ L_pg, KỂ CẢ khi hội tụ". Khi
  policy gần tối ưu (nhất là với gae>0) L_pg → rất nhỏ → β cố định/cao làm β·H lấn át
  → policy trôi về NGẪU NHIÊN → un-learn QoS. Harness in cờ "⚠ ENTROPY-DOMINATED".
  → β phải anneal về sàn; resume policy tốt thì β thấp + anneal nhanh.

SPSA (lượng tử): ước lượng gradient QC bằng n_reps nhiễu (rẻ hơn parameter-shift).
  nq lớn → SPSA nhiễu hơn → λ dễ "đóng băng" ở init (không học). Tăng spsa_n_reps
  cho gradient sạch hơn.

CURRICULUM (R_LoS): tăng độ khó hình học dần 0.2→0.3→0.4→0.45→0.5, resume mỗi bước.
  R_LoS lớn = user/IRS xa hơn = link yếu hơn = QoS khó hơn. ỔN ĐỊNH ở bước hiện tại
  RỒI mới lên — lên sớm = mang lỗi sang env khó hơn.

CHỌN MODEL TỐT NHẤT: theo QoS từ checkpoints/, KHÔNG dùng thư mục best/ (best lưu theo
  REWARD, không cùng thang giữa các λ_D, và đỉnh reward thường là ep sớm chứ không phải
  đỉnh QoS).

          kẹt DƯỚI cả baseline Direct-only. KHẮC PHỤC: HẠ λ_D mạnh (vd 3.0→1.5-2.0) để
          return học được; nếu đang fresh thì resume từ checkpoint để critic warm-up
          re-fit V về thang mới. Dấu hiệu nhận biết: qp/step ≫ R̄, explVar kẹt ~0.1,
          QoS < Direct-only baseline.
    QUY TẮC CALIBRATE λ_D: nhắm theo baseline TẦM THƯỜNG TỐT NHẤT (gồm cả Direct-only,
    KHÔNG chỉ All-IRS). Trivial-baseline ~100% QoS → giữ λ_D THẤP. Baseline thấp/khó →
    λ_D cao hơn. (result_1 Case 2: Direct-only=100% mà dùng λ_D=3.0 → critic mù, kẹt 54%.)

[J] DIRECT-ATTRACTOR / IRS CHICKEN-AND-EGG — agent bỏ dần IRS, kẹt local optimum
    "all-direct"; phase head idle (đói data) → không học pha → IRS mãi tệ → càng tránh.
    Dấu hiệu: behaviour `dir` tăng dần về K, `irs=[..]` tụt về ~0, ent phase kẹt CAO
    (~50-60% max) dù IRS không dùng; QoS có thể vẫn thấp vì power/IRS path chưa học.
    Nguyên nhân (vòng luẩn quẩn): pha KHỞI TẠO NGẪU NHIÊN (env random_phi) → IRS link
    tệ lúc đầu → critic credit "IRS xấu" → agent chọn direct → phase head ít sample IRS
    → pha mãi ngẫu nhiên. Bị che bởi D_k THẤP: direct dù bị block vẫn vượt D_k (QoS
    trivial) nên IRS "không cần" cho QoS — nhưng IRS-aligned cho SUM-RATE cao hơn nhiều
    cho user bị block (beamforming gain ∝ N²) → reward tối ưu THỰC SỰ dùng IRS.
    ⚠️ CAVEAT PROBE: baseline "All-IRS %" dùng PHA NGẪU NHIÊN (AllIRSPolicy KHÔNG set
    phase_idx) → là lower-bound BẤT CÔNG, KHÔNG phải tiềm năng IRS. ĐỪNG kết luận "IRS
    tệ hơn direct" từ nó. Với pha căn chỉnh, IRS thường vượt direct (nhất là user bị block).
    → DÙNG probe_irs_vs_direct.py (so |h_IRS|_opt vs |h_direct| per user) để biết IRS-usage
      cao có optimal không. PHÁT HIỆN (Case2 R_LoS=0.2): IRS-opt thắng direct 100% ở user
      BỊ BLOCK (+38dB) nhưng chỉ MARGINAL ở non-block (+1.5dB). Pha ngẫu nhiên yếu ~14dB
      (|Σφ| N→√N) → block vẫn thắng (dùng IRS), non-block lật sang THUA (giữ direct).
      → IRS/K ≈ block-count là HỢP LÝ khi pha xấu. LEVER đẩy IRS-usage cao (route non-block)
      = CHẤT LƯỢNG PHA, KHÔNG phải assignment/readout. Thứ tự: critic→pha→readout.
    KHẮC PHỤC: (1) [mạnh nhất] PHASE WARM-START — pretrain pha về căn chỉnh kênh
    (max |g_SR·Φ·g_RU|) thay random init → agent thử IRS thì IRS đã tốt sẵn → phá vòng.
    (2) giữ exploration q-head (IRS-select) lâu hơn để phase có data (coi chừng [C]).
    (3) curriculum R_LoS tăng dần → IRS cần thêm từng chút → agent thích nghi dần,
    tránh nhảy vào ramp khó với IRS path trắng. KHÔNG resume từ checkpoint đã "tránh IRS"
    (warm-up sẽ re-fit critic theo policy direct-only → đóng đinh local optimum).
    (result_1 Case 2 @R_LoS=0.2: Blk~6/10 nhưng agent drift dir 4→8, irs→~1, phase idle.)
