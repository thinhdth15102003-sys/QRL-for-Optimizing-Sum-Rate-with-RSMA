#    PHẢI calibrate qua feasibility probe ở từng ramp.

# ── QUY TRÌNH CALIBRATE MỖI RAMP (quan trọng hơn con số khởi điểm) ──────────────
#   1. Chạy → đọc dòng feasibility probe: "Servable %" và "All-IRS QoS %".
#   2. Nếu Servable < ~95% → ĐỪNG đẩy λ_D cao (phạt user bất khả thi, lỗi [I]); giữ vừa.
#   3. So All-IRS baseline với Case 1 cùng R_LoS:
#        - Case 2 CAO HƠN nhiều (dễ hơn nhờ P↑) → HẠ λ_D ~0.5–1 so với bảng trên.
#        - Thấp hơn → giữ/tăng.
#   4. Theo triệu chứng mà tinh chỉnh:
#        - plateau [D] (reward phẳng, best không vượt)      → TĂNG λ_D
#        - reward âm sâu ngay đầu / QoS trần thấp [I]        → GIẢM λ_D

# ── KIM CHỈ NAM: nhắm λ_D theo All-IRS baseline (map từ Case 1) ─────────────────
#   All-IRS QoS ≈ 70%  → λ_D ≈ 3.0
#   All-IRS QoS ≈ 65%  → λ_D ≈ 3.5–4.0
#   All-IRS QoS ≈ 60%  → λ_D ≈ 4.5
#   (Case 1 ref: 0.2→70%, 0.3→63%, 0.4→60%, 0.45→59%, 0.5→59%, servable luôn 98.9%.)

# ── LÝ THUYẾT: λ_D KHÔNG cần scale theo K ──────────────────────────────────────
#   λ_D là trade-off PER-USER (QoS vs sum-rate biên). Penalty là Σ trên K user nhưng
#   cân bằng per-user độc lập K; advantage được chuẩn hóa per-rollout. → schedule trên
#   hợp lý cho K=10; chỉ calibrate theo ĐỘ KHÓ (probe), KHÔNG nhân theo số user.
